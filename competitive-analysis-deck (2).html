Mailchimp

The Benchmark for Approachable Email Building

Gold standard for drag-and-drop simplicity

Slide 3 · Mailchimp

Overview

Who it's for

  • Small-to-medium businesses
  • Non-technical marketers
  • Users who want to "just send an email" without engineering support

Why it matters for us

  • "Content Studio" concept aligns with our Library idea
  • Property managers' mental model likely shaped by tools like this
  • Sets the baseline for simplicity expectations
Key insight

Our users will expect this level of simplicity as the baseline experience.

Slide 4 · Mailchimp

Content Block Model

  • Drag-and-drop blocks – text, image, button, divider, etc.
  • "Saved Content Blocks" feature lets users save any block for reuse
  • Saved blocks appear in a dedicated sidebar panel during editing
Key distinction: Mailchimp's saved blocks are copy-paste reusable, not linked reusable. Each instance is independent after insertion.
Design implication

We need to go beyond this—our blocks must stay linked so edits propagate. But the discovery UI (sidebar with previews) is worth stealing.

Slide 5 · Mailchimp

Personalization & Rules UI

  • "Conditional Merge Tags" for simple show/hide
  • Segmentation happens at the audience level, not block level
  • No native "show this block only to segment X" toggle in the visual editor
  • Advanced users must write merge tag code manually
*|IF:GROUPINGS:GroupName=VIP|*
  VIP content here
*|END:IF|*
Mapped to our use cases

Default vs Group PreArrival: Would require separate campaigns or manual merge tags
Club-only content: No clean visual way to do this per-block

Key insight

Mailchimp's personalization is too code-heavy for property managers. We need a visual alternative.

Slide 6 · Mailchimp

What's Great / What's Not

✓ Patterns worth stealing

  • Content block sidebar with visual previews
  • "Save as block" action directly from the editor
  • Clean, uncluttered canvas—blocks feel like physical objects
  • Drag-to-reorder is intuitive and smooth
  • "Content Studio" as a standalone library hub

✗ Too complex or not suitable

  • Conditional content requires merge tag code
  • Saved blocks aren't truly linked (edits don't sync)
  • No visibility into "where else is this block used?"
  • No per-block audience targeting in visual editor
  • AI features are surface-level, not integrated
Klaviyo

Best-in-Class Block-Level Conditional Logic

E-commerce leader proving visual personalization works

Slide 7 · Klaviyo

Overview

Who it's for

  • E-commerce marketers (Shopify, BigCommerce)
  • Users comfortable with segmentation
  • More technically ambitious than Mailchimp users

Why it matters for us

  • Best-in-class "Show/Hide" conditional logic at block level
  • Profile property targeting maps well to guest attributes
  • Strong model for "if guest has X, show Y"
Key insight

Klaviyo proves block-level conditional visibility can be done visually—no code required.

Slide 8 · Klaviyo

Content Block Model

  • Drag-and-drop blocks similar to Mailchimp
  • "Universal Content" blocks = truly linked, reusable content
  • When you update a Universal Content block, it updates everywhere it's used
  • Universal Content appears in a dedicated library section
Key distinction: Klaviyo does have linked blocks (Universal Content). This is closer to what we need.
Design implication

Universal Content is the model we should follow. The "Edit across all" vs "Unlink" choice is exactly what we prototyped.

Slide 9 · Klaviyo

Personalization & Rules UI

  • "Show/Hide" toggle on every block in the sidebar
  • Rules use a visual query builder: Property + Operator + Value
  • Example: "Show if VIP_Status equals true"
  • Can stack multiple conditions with AND/OR logic
  • Preview lets you test with different profile properties
Mapped to our use cases

Default vs Group PreArrival: Add condition guest_type equals Group to show Group Welcome Letter
Club-only content: Add condition club_eligible equals true AND hotel_has_club equals true

Key insight

This is almost exactly what we need. The visual condition builder + preview is the pattern.

Slide 10 · Klaviyo

What's Great / What's Not

✓ Patterns worth stealing

  • Universal Content = linked blocks that sync across emails
  • Visual Show/Hide conditions on any block (no code)
  • "Preview as profile" to test personalization
  • Clean condition builder with dropdowns
  • "Edit all" vs "Unlink" choice when editing shared blocks

✗ Too complex or not suitable

  • E-commerce focus means UI language doesn't translate
  • Condition builder can get complex with many AND/OR branches
  • No usage count shown ("used in X emails")
  • Flow-based automation is overkill for pre-arrival emails
  • Segment targeting lives separately (two mental models)
HubSpot

The "Smart Content" Pioneer

One block, multiple variants per audience

Slide 11 · HubSpot

Overview

Who it's for

  • B2B marketers with CRM integration needs
  • Teams wanting personalization tied to contact properties
  • Users who need "smart content" for website + email

Why it matters for us

  • "Smart Content" = same block, different versions per audience
  • Pioneered the "default + variant" model within a single block
  • Strong metaphor for "Club guests see this, everyone else sees that"
Key insight

HubSpot's "Smart Content" is the closest analog to our "block variants by segment" need.

Slide 12 · HubSpot

Content Block Model

  • "Modules" (their term for blocks) can be saved to a library
  • "Global Modules" are truly linked—edit once, update everywhere
  • "Local Modules" are one-off instances
  • Clear visual distinction: Global modules have a special icon/badge
Key distinction: HubSpot explicitly differentiates Global vs Local—and lets you convert between them.
Design implication

The Global vs Local distinction + conversion flow is exactly what we need for "keep shared" vs "create one-off."

Slide 13 · HubSpot

Personalization & Rules UI (Smart Content)

How Smart Content works

  1. Click "Make smart" on any module
  2. Choose the criteria: Contact list, Lifecycle stage, Country, Device, etc.
  3. Create variant versions of that module for each segment
  4. Default version shows for everyone who doesn't match a rule
Mapped to our use cases

Default vs Group PreArrival: Welcome Letter block → Make smart → "Guest Type = Group" variant
Club-only content: Club Detail block → Make smart → "Club Eligible = Yes" (and hide default)

Key insight

The "tabs for variants" UI is very clear. Users see all versions in one place.

Slide 14 · HubSpot

What's Great / What's Not

✓ Patterns worth stealing

  • "Make smart" as an explicit action—progressive disclosure
  • Variant tabs within a single block (Default + Segment A + B)
  • Global vs Local distinction with conversion option
  • Warning when editing global module
  • Preview toggle to see each variant

✗ Too complex or not suitable

  • Smart Content criteria are CRM-centric—need translation
  • Setup requires pre-defined contact properties
  • UI can feel cluttered with 4+ variants
  • No visual indicator on canvas that block has smart rules
  • "Global module" editing opens a separate editor
Beefree

Design System Approach to Email

Strong model for brand-controlled templates with local flexibility

Slide 15 · Beefree

Overview

Who it's for

  • Agencies building templates for clients
  • Users who want design-system consistency
  • Non-technical designers needing brand control

Why it matters for us

  • "Saved Rows" = reusable content blocks that sync
  • "Content Locking" = control what editors can/can't change
  • Model for brand-controlled templates with local flexibility
Key insight

The "locked vs editable" concept could help property managers understand what they can customize vs what's centrally controlled.

Slide 16 · Beefree

Content Block Model

  • "Rows" are the primary unit (can contain multiple content blocks)
  • "Saved Rows" are reusable and synced across templates
  • "Synced Rows" (newer feature) = true linked content, updates propagate
  • Row library has categories, tags, and search
Key distinction: Beefree separates "Saved" (copy) from "Synced" (linked)—explicit terminology.
Design implication

Explicit "Synced" terminology is clearer than HubSpot's "Global." The warning modal is exactly what we need.

Slide 17 · Beefree

What's Great / What's Not

✓ Patterns worth stealing

  • "Synced Row" as explicit terminology
  • Row library with usage counts
  • Content locking for brand control
  • Clean, minimal condition builder
  • Category + tag system for organizing content

✗ Too complex or not suitable

  • Personalization depends on connected ESP
  • No "create variant for segment" within a row
  • Row-level granularity may be too coarse
  • No preview-as-segment feature
  • Agency-focused terminology
Salesforce Marketing Cloud

Enterprise Personalization at Scale

The ceiling of what's possible (and complexity to avoid)

Slide 18 · SFMC

Overview

Who it's for

  • Enterprise marketers with complex needs
  • Teams with developer support for AMPscript
  • Organizations already in Salesforce ecosystem

Why it matters for us

  • "Content Builder" = enterprise-grade reusable content
  • "Dynamic Content" = advanced rule-based variants
  • Learn from their power, design for 10% complexity
Slide 19 · SFMC

Content Block Model

  • "Content Blocks" live in Content Builder library
  • Blocks can be nested, versioned, and organized in folders
  • "Reference Content" = linked blocks that sync
  • "Copy Content" = independent instances
Key distinction: Explicit choice when inserting: "Reference" vs "Copy." Very clear mental model.
Design implication

"Reference vs Copy" choice at insertion is worth considering—makes the decision explicit upfront.

Slide 20 · SFMC

What's Great / What's Not

✓ Patterns worth stealing

  • "Reference vs Copy" choice at insertion time
  • Block usage reports ("used in 47 emails")
  • Folder + search organization
  • Preview with simulated subscriber attributes
  • Versioning / approval workflows

✗ Too complex or not suitable

  • Dynamic Content rule UI requires training
  • AMPscript for advanced logic—code, not visual
  • Deeply nested folders create discoverability issues
  • Enterprise pricing and setup
  • Assumes dedicated "email developer" role

Patterns to Steal

Cross-tool patterns especially good for our use cases

Slide 21 · Patterns

Pattern: Linked Blocks with Usage Visibility

What it is

A reusable content block that syncs across all emails where it's used, with clear visibility into usage count and locations.

Who does it well

Klaviyo Beefree SFMC
Design implication

Our Library sidebar should show "Shared · 4 emails" and offer a "See where used" link.

Slide 22 · Patterns

Pattern: Visual Show/Hide Conditions per Block

What it is

A visual interface for setting display conditions on individual blocks, without code.

Who does it well

Klaviyo Beefree HubSpot
Design implication

Our block edit panel should have a "Who should see this?" section with checkboxes/dropdowns, not code.

Slide 23 · Patterns

Pattern: Explicit Shared vs Local Choice

What it is

A clear decision point where users choose whether edits affect all instances or just this one.

Who does it well

Klaviyo HubSpot Beefree
Design implication

Our prototype's "Update everywhere" vs "Change only for this email" radio buttons are the right pattern. Add a confirmation warning for global updates.

Slide 24 · Patterns

Pattern: Variant Tabs Within One Block

What it is

Showing all segment-specific versions of a block in a single UI, as tabs or panels.

Who does it well

HubSpot SFMC
Design implication

Consider adding variant tabs inside the block edit panel when multiple audience rules exist.

Slide 25

Gaps & Opportunities

Where none of these tools do a great job—especially for property managers.

Gap 1: Explaining shared vs overridden

Most tools show an icon/badge but don't explain implications.

Contextual microcopy ("Editing this will update 4 other emails")

Gap 2: Showing who sees what

Preview modes exist but require manual toggling.

Summary view: "Group guests → Group Welcome; All others → Default"

Gap 3: Making personalization feel safe

Condition builders can create logic errors.

Plain-language summaries, preview-before-save, easy rollback

Gap 4: Property-specific context

None understand hospitality concepts.

Pre-built conditions like "Guest is club-eligible"

Gap 5: Cross-email visibility

Hard to see what content emails have in common. No side-by-side comparison view.

Email comparison mode showing shared vs different blocks
Slide 26

Design Principles for Our Email Studio UI

  1. Linked by default, override by choice

    Blocks stay connected unless the user explicitly creates a local version.

  2. Plain language rules, not logic gates

    Express conditions as "Show to: Group guests" not "IF guest_type = 'Group'".

  3. Show the blast radius

    Before editing shared blocks, show exactly which emails will be affected.

  4. One block, visible variants

    Show all segment versions in one panel. Don't hide variants behind separate flows.

  5. Preview is mandatory, not optional

    "See as: Default Guest | Group Guest | Club Guest" toggle always visible.

  6. Library as source of truth

    The Content Block Library is the canonical home for all shared content.

  7. Progressive disclosure for power features

    Simple use cases require zero rule configuration. Targeting appears when needed.

  8. Safety nets everywhere

    Undo, version history, "Are you sure?" for global changes.
